## 一、执行规范（摘自开发策略文档，P2 全程强制）

### 1.1 PRD 强制遵循

P2 所有实现必须严格对照以下 PRD：

| PRD 文件 | 覆盖范围 |
|----------|---------|
| `01-data-model.md` | Employee / PayrollInput / PaySlip / SocialConfig 类型定义 |
| `05-mod-payroll.md §二` | 单人计算公式（6 步）+ 全员汇总规则 + 公司筛选 |
| `05-mod-payroll.md §一` | 页面 UI 需求（按员工 + 明细表） |

### 1.2 代码质量准则

根据开发策略文档 v3.6 §1.5：

- 不设硬性行数上限，但每个文件聚焦**单一职责**
- 文件超过 300 行或职责模糊时，必须考虑拆分
- 代码应简洁优雅、模块化、可维护性强、解读性强
- 金额计算必须使用 `decimal.js`，禁止原生浮点运算
- 所有用户可见文字走 i18n（`t('key')`），禁止硬编码中文
- TypeScript strict 模式，禁止 `any` 类型
- 每个 service 函数必须是纯函数（零副作用，可独立测试）
- 组件 props 必须有明确的 TypeScript 接口定义
- 所有测试文件放在 `tests/` 目录下

### 1.3 AI 模型分级

| 环境 | 模型 | 说明 |
|------|------|------|
| Codex | GPT-5.3-codex xhigh | 所有子阶段的开发执行 |
| Claude Code CLI | Opus 4.6 | 代码审查、修复、文档更新 |

### 1.4 测试策略（策略文档 §4）

- 每个 service 函数写完后立即写对应单元测试
- 测试环境隔离：`APP_ENV=test` + `TEST_DB_PATH` 指向临时目录
- ABI 切换：`npm run test` 使用 Node ABI，`npm run test:e2e` 使用 Electron ABI
- 里程碑三件套：raw JSON + case map 对账 + XLSX 报告

## 七、风险与缓解

| 风险 | 缓解措施 |

## 八、架构决策（P2 锁定）

| 决策 | 选择 | 理由 |
|------|------|------|
| calculator 输入 | `(Employee, PayrollInput, SocialConfig) → PaySlip` | 纯函数，零副作用，最大可测试性 |
| aggregator 输入 | `(PaySlip[], filterCompany?) → AggregateResult` | 与 calculator 解耦，P3 凭证直接消费 |
| payroll 持久化 | JSON payload 存入已有 `payroll_inputs` / `payroll_results` 表 | 复用 P1 schema，无需新 migration |
| store 粒度 | 单一 `payroll-store`，按 month 管理 | 避免过度拆分，月度数据天然一体 |
| PDF/CSV 导出 | P2 不做，推迟到 P2.5 或 P3 前置 | 聚焦核心计算 + UI，避免分散精力 |
