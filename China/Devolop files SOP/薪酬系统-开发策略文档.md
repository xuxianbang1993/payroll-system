## 〇、AI 模型分级选用策略（最高优先级）

本章节定义不同开发环境和任务复杂度下的 AI 模型选用规则。**所有开发操作必须先确认运行环境，再选用对应模型。**

### 0.1 Codex 环境（最优先）

| 环境 | 默认模型 | 模式 | 说明 |
|------|----------|------|------|
| **Codex** | GPT-5.3-codex | **xhigh** | Codex 环境下统一使用此配置，不做难度分级 |

> ⚠️ 在 Codex 环境运行时，无论任务复杂度如何，一律使用 `GPT-5.3-codex xhigh` 模式。此规则优先级高于下方 Claude Code CLI 分级。

### 0.2 Claude Code CLI 环境（按任务复杂度分级）

| 难度 | 模型 | 适用场景 |
|------|------|----------|
| **简单** | Haiku 4.5 | 单文件小改、格式调整、简单重命名、i18n key 补充、文档微调 |
| **中等** | Sonnet 4.5 Thinking | 单模块功能开发、组件实现、store 编写、单元测试编写、bug 定位与修复 |
| **高难度** | Opus 4.6 | 跨模块架构设计、核心业务逻辑（calculator/voucherGenerator）、E2E 流程编排、数据库迁移、多文件联动重构 |

判断依据：

```
简单任务（自动调用 Haiku 4.5）：
  - 改动范围 ≤ 1 个文件
  - 不涉及业务逻辑
  - 无需联网查证
  - 例：补一个 i18n key、修一个 typo、调一个样式

中等任务（Sonnet 4.5 Thinking）：
  - 改动范围 2-5 个文件
  - 涉及单模块业务逻辑
  - 可能需要联网查证 1-2 个技术点
  - 例：实现一个页面组件、编写一组单元测试、修复一个中等 bug

高难度任务（Opus 4.6）：
  - 改动范围 > 5 个文件或涉及跨层联动
  - 涉及核心计算公式、凭证生成、数据库 schema 变更
  - 需要联网查证多个技术点
  - 例：实现 calculator.ts 全套计算逻辑、设计 repository 抽象、编排 E2E 测试流程
```

## ⛔ 一、AI 行为红线（必须遵守）

本章节定义 AI 在开发过程中**不可违反**的行为边界。

### 1.1 不可自导自演

```
❌ 禁止行为：
- 不确定某个 API 用法时，凭记忆编造代码
- 不确定某个库的配置方式时，"猜测"一个看起来合理的写法
- 遇到报错时，未查证就连续尝试多种"可能的修复"
- 编造不存在的组件属性、函数参数、配置项

✅ 正确行为：
- 承认"我不确定这个 API 的具体用法"
- 通过 MCP 联网搜索官方文档
- 基于官方文档的实际示例编写代码
```

### 1.2 必须联网查证的场景

以下场景**必须**通过 MCP 工具联网搜索官方文档，不可凭记忆作答：

| 场景 | 搜索目标 |
|------|----------|
| Electron API 调用（IPC、窗口管理、菜单等） | electronjs.org 官方文档 |
| shadcn/ui 组件用法或自定义主题 | ui.shadcn.com 官方文档 |
| Tailwind CSS 工具类或配置 | tailwindcss.com 官方文档 |
| TanStack Table 配置和 API | tanstack.com/table 官方文档 |

| electron-store（轻配置层） | npm 文档 / GitHub README |
| electron-builder 打包配置（dmg / exe） | electron.build 官方文档 |
| i18next / react-i18next 配置 | i18next.com 官方文档 |
| jsPDF / html2canvas API | 对应 npm / GitHub 文档 |
| xlsx (SheetJS) 导入导出 API | docs.sheetjs.com |

### 1.3 联网失败时的处理

```
如果 MCP 联网搜索失败或无法访问目标文档：
  1. 立即停止当前任务
  2. 明确告知用户："联网搜索失败，无法查证 [具体内容]，请检查网络或手动提供文档"
  3. 不可退而求其次地"凭记忆写一版试试"
  4. 等待用户提供文档或确认后再继续
```

### 1.4 PRD 强制遵循

```
所有功能实现必须严格对照 PRD 文件：
  1. 开始任何模块开发前，必须先读取对应的 PRD 文件
  2. 字段名、类型、计算公式必须与 PRD 01-data-model.md 完全一致
  3. UI 结构必须与 PRD 中描述的页面布局一致
  4. 不可"优化"或"改进" PRD 中明确定义的业务逻辑（如有建议，先提出再等确认）
  5. 每完成一个功能，对照 PRD 对应章节逐项自检
```

### 1.5 代码质量红线

```
- 单个文件不超过 200 行（超出必须拆分）
- 所有用户可见文字必须走 i18n（t('key')），禁止硬编码中文
- 金额计算必须使用 decimal.js，禁止原生浮点运算
- TypeScript strict 模式，禁止 any 类型（极少数必要场景需注释说明理由）
- 每个 service 函数必须是纯函数（零副作用，可独立测试）
- 组件 props 必须有明确的 TypeScript 接口定义
```

**字体规范**（延续原系统设计）：

| 用途 | 字体 | 说明 |
|------|------|------|

## 四、测试策略

### 4.1 测试分层

| 层级 | 工具 | 覆盖范围 | 运行时机 |
|------|------|----------|----------|
| **单元测试** | Vitest | services/、utils/、stores/ | 每次提交前 |
| **组件测试** | Vitest + React Testing Library | components/ 交互逻辑 | 每次提交前 |
| **E2E 测试** | Playwright | 完整业务流程 | 每个 Phase 完成后 |

### 4.1.1 测试环境隔离开关（SQLite 必须遵守）

| 开关 | 取值 | 作用 | 默认值 |
|------|------|------|--------|
| `APP_ENV` | `prod` / `test` | 控制运行模式与破坏性能力开关 | `prod` |
| `TEST_DB_PATH` | 绝对路径 | test 模式下指定 SQLite 测试库位置 | 空（自动临时路径） |
| `READ_SOURCE` | `legacy` / `sqlite` | 控制读取来源（迁移灰度与回滚） | `legacy`（迁移期） |
| `WRITE_MODE` | `legacy` / `dual` / `sqlite` | 控制写入策略（单写旧库/双写/单写新库） | `legacy`（迁移期） |

隔离规则（强制）：

1. 功能测试、E2E、数据库重置测试必须使用 `APP_ENV=test`
2. `resetDatabase()` 仅允许在 `APP_ENV=test` 调用，非 test 模式必须 fail-fast
3. 每轮测试必须使用独立测试库路径，运行后清理，不可复用正式库
4. 测试启动日志必须打印 `APP_ENV`、DB 路径、`READ_SOURCE`、`WRITE_MODE`
5. 涉及 `better-sqlite3` 的测试必须匹配运行时 ABI：`npm run test` 使用 Node ABI、`npm run test:e2e` 使用 Electron ABI

### 4.2 测试治理权威源（Single Source of Truth）

测试治理规则只在本文档维护，`China/test/` 目录只承载映射、样例数据和报告产物，不再维护第二套规则文本。

1. `raw JSON`：测试原始结果（Vitest / Playwright）

#### B. 正式里程碑（P0 / P1 / P2 / P3 / P4 阶段关门）
